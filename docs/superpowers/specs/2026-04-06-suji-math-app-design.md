# claude-suji: 수지의 수학 학습 파트너 — 설계 문서

**날짜**: 2026-04-06
**작성자**: Claude + 아빠 (브레인스토밍 세션)
**대상 사용자**: 수지 (중2, 중1 수학 기초 약함)
**상태**: 설계 확정

---

## 1. 프로젝트 배경

### 1.1 사용자 상황 (수지)
- 현재 중2. 작년 중1 수학 시험 28점 (학년 평균 57점).
- 개념을 어느 정도 알지만 **문제를 보고 어떻게 시작해야 할지 모름**.
- **2개 이상 개념이 섞이면** 접근 못 함.
- 알고 있는 개념이 나와도 **그게 그 개념인지 식별 못 함**.
- 응용문제는 손도 못 댐.
- 웹툰을 좋아함.

### 1.2 프로젝트 목표
수지의 "개념 식별 약점"을 훈련시키는 **개인용 수학 학습 웹앱**. 중1 1학기 재정비부터 시작하여 점진적으로 확장.

### 1.3 가치 원칙
- **"수지는 친구와 놀듯 배우고, 아빠는 뒤에서 지원만"**
- 감시 감독 기능 배제 (수지의 자율성/자존감 보호)
- 수지 피드백 기반 반복 개선

## 2. MVP 정의 (Phase 1)

### 2.1 학습 기능 우선순위
| 단계 | 기능 | Phase |
|------|------|-------|
| B | 개념 식별 훈련 | **MVP** |
| A | 시작법 가이드 | Phase 3 |
| C | 개념 조합 훈련 | Phase 5 |

**MVP는 B 단계만 구현**한다. A, C는 로드맵에 있으나 초기 범위에서 제외.

### 2.2 콘텐츠 범위
- **중1 1학기 (1-3단원)**: 수와 연산, 문자와 식, 그래프와 비례
- 개념 수: 약 20-25개 (추출 후 확정)
- 문제 수: 개념당 **Lv1 + Lv2만** × 각 10문제 = 약 400-500문제 (MVP는 B단계 = 개념 식별만)
- 문제별 웹툰 스크립트 1개
- Lv3(혼합)/Lv4(응용)은 스키마에만 존재, MVP 콘텐츠 없음

### 2.3 기능 목록
- 수지/아빠 로그인 (2인 계정, role 분리)
- 데일리 미션 (10문제) + 자유 연습
- 개념 식별 훈련: 문제 → 개념 멀티셀렉트 → 즉시 피드백
- 텍스트 기반 웹툰 대화체 해설
- 숙달도 추적 (스키마는 Lv1-Lv4 지원, MVP에서 수지는 Lv1→Lv2까지만 도달 가능)
- 스트릭 카운트
- 아빠 관리자 대시보드 (진도, 문제 관리, 개념 관리, 스티커 주기, CSV 내보내기)
- 수지 "이 문제 이상해요" 신고 버튼
- Python 콘텐츠 파이프라인 (고정 문제 방식, CLI 실행)
- 3단 할루시네이션 방어

### 2.4 성공 기준
**수지 관점:**
- 매주 5일 이상 데일리 미션 완료
- 2주 내 Lv1 개념 5개 이상 마스터
- 피드백: "재미있다" 이상

**기술 관점:**
- 페이지 로드 1초 이내 (모바일 3G)
- Vercel/Supabase 무료 티어 내 운영
- 보안 침해 0건

## 3. 시스템 아키텍처

### 3.1 3-계층 구조

```
┌─────────────────────────────────────────────────┐
│  Content Pipeline (Python, 오프라인 배치)        │
│  PDF(MathFile/중1) → 개념추출 → 아빠 검토        │
│         ↓                                        │
│  4-페르소나 창작 → sympy 검증 → 3-페르소나 리뷰  │
│         ↓                                        │
│  전원 PASS → DB 저장                             │
└─────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────┐
│  Supabase (Postgres + Auth + RLS)              │
└─────────────────────────────────────────────────┘
              ↑
┌─────────────────────────────────────────────────┐
│  Next.js Web App (Vercel)                       │
│  수지: 데일리미션 + 자유연습                      │
│  아빠: 관리자 대시보드                            │
└─────────────────────────────────────────────────┘
```

### 3.2 설계 원칙
1. **콘텐츠 파이프라인 = 오프라인**: 문제 생성은 비싸고 느림. DB에 채워두고 웹앱은 읽기만.
2. **다과목 확장 대비**: `subject_id` 최상위. 국어/영어/과학 추가 시 스키마 유지.
3. **할루시네이션 3단 방어**: sympy + 3-페르소나 + 아빠 샘플링.
4. **데이터 누적**: 모든 시도/정답률 기록. Phase 6(적응형) 재료.

## 4. 데이터 모델

### 4.1 계층 구조

```
subject (수학)
  └─ grade (중1)
       └─ unit (1. 수와 연산)
            └─ sub_unit (1-1. 정수와 유리수)
                 └─ concept (절댓값)
                      └─ problem
```

### 4.2 테이블 정의

**콘텐츠 테이블 (Python 워커가 씀)**

```
subjects          id, name, created_at
grades            id, subject_id, name, order
units             id, grade_id, name, order
sub_units         id, unit_id, name, order
concepts          id, sub_unit_id, name, description, order
problems          id, concept_primary_id, difficulty(1-4), body, answer,
                  status("published"/"draft"/"rejected"), source_trace(JSONB),
                  sympy_verified(bool), created_at
problem_concepts  problem_id, concept_id  -- M:N 다중 태깅
solutions         problem_id, steps(JSONB)
webtoon_scripts   problem_id, script(JSONB)  -- [{speaker, text}, {scene, text}]
```

**학습 테이블 (웹앱이 씀)**

```
users             id(supabase), role("student"/"admin"), nickname, created_at
sessions          id, user_id, started_at, ended_at, mode("daily"/"free")
attempts          id, session_id, user_id, problem_id,
                  selected_concepts(JSONB), is_correct, time_ms, attempted_at
concept_mastery   user_id, concept_id, level(1-4), accuracy_recent10,
                  attempts_total, last_updated, downgrade_warnings
rewards           id, user_id, kind("sticker"/"praise"), content, granted_at, granted_by
problem_reports   id, user_id, problem_id, note, status("open"/"resolved"), reported_at
```

**감사 로그**

```
problem_reviews   id, problem_id, stage("sympy"/"persona"), persona_name,
                  verdict("pass"/"fail"), reason, reviewed_at
```

### 4.3 핵심 설계 포인트
- `problem_concepts` M:N → 혼합 문제(C단계) 대비
- `source_trace` JSONB → 저작권 감사용 (원문 X, 참고 메타만)
- `subject_id` 최상위 → 다과목 확장
- `concept_mastery.level` → 수지 숙달도 레벨

### 4.4 나중에 추가할 테이블
- `problem_templates` (Phase 4, 파라메트릭 문제)
- `characters` (Phase 7, 웹툰 캐릭터)
- `review_schedules` (Phase 6, SRS)
- `learning_paths` (Phase 3, A단계 가이드)

## 5. 숙달도 규칙 (Mastery)

### 5.1 난이도 레벨 (중학 수준 상한)

| Lv | 이름 | 특징 | MVP 사용 |
|----|------|------|----------|
| 1 | 기본 인식 | 개념 표현 명확 | ✅ |
| 2 | 변형 표현 | 표현 살짝 변경 | ✅ |
| 3 | 혼합 | 2개 개념 섞임 | C단계 |
| 4 | 응용 | 문장제, 중학 상한 | A단계 이후 |

**중학 수준 상한 기준:**
- ❌ 고교 선행 개념 금지
- ❌ 경시대회/심화 수준 금지
- ✅ 중학 교과과정 안에서 응용/심화
- ✅ 검증팀 "출판관계자" 페르소나가 이 기준으로 최종 필터링

### 5.2 숙달 판정 규칙 (하이브리드)
- **업그레이드 조건**: 정답률 80% AND 최근 10문제 중 8문제 정답 AND 최소 15문제 시도
- **다운그레이드 조건**: 현재 레벨 최근 5문제 중 3문제 틀림 → 한 단계 내림 + 복습
- **시간 기준 없음** (수지마다 속도 다름)
- **자동 판정** (아빠 수동 개입 없음)

## 6. 콘텐츠 파이프라인 (Python 워커)

### 6.1 전체 흐름

```
Step 1: PDF → 개념 추출 (AI 초안)
Step 2: 아빠 개념 검토 (CLI)
Step 3: 문제 창작 (4-페르소나)
Step 4: sympy 수치 검증 (가능한 경우)
Step 5: 3-페르소나 검증팀 리뷰
Step 6: 전원 PASS → DB 저장
Step 7: 웹툰 스크립트 생성
```

### 6.2 페르소나 역할

**창작팀 (Step 3, 4인)**
- 교육학자: 수지 수준 맞춤, 스캐폴딩, 혼란 유발 표현 피함
- 일타강사: 핵심 개념 표현 명확, 수식 정확
- 심리학자: 좌절 방지, 난이도 점진적
- 웹툰디자이너: 일상 상황/캐릭터 등장 여지

**검증팀 (Step 5, 3인, 독립 평가)**
- 출판관계자: 저작권 이슈, 기존 문제와 유사도, **중학 수준 상한 유지**
- 학습전문가: 개념 태깅 정확성, 난이도 적절성, 교과과정 부합
- 일타강사: 풀이법 합리성, 표현 모호성

### 6.3 할루시네이션 방어
1. **sympy 수치 검증** (기계적 진실, 가능한 문제)
2. **3-페르소나 독립 리뷰** (전원 PASS 필수)
3. **source_trace 기록** (참고 PDF 메타만, 원문 저장 X)
4. **아빠 샘플링** (원할 때 대시보드에서 확인)

### 6.4 재시도 정책
- sympy 검증 실패 → Step 3 재시도 (최대 3회)
- 페르소나 1명이라도 FAIL → 실패 이유를 프롬프트에 반영 후 재시도 (최대 3회)
- 3회 실패 시 폐기 + 로그

### 6.5 문제 부족 대응 (MVP)
- 초기 인벤토리: 개념당 **Lv1 + Lv2 × 각 10문제** 고정
- **자동 보충 트리거**: "미본 문제 < 30%" 또는 "약점 정답률 < 60%" → 해당 개념 추가 생성
- Phase 4에서 파라메트릭 템플릿 도입 예정 (무한 풀)

### 6.6 LLM 선택
- **창작 + 검증**: Claude Sonnet 4.6 (claude-sonnet-4-6)
- **샘플 검수**: Claude Opus 4.6 (claude-opus-4-6)

### 6.7 예상 초기 비용
- 20 개념 × 2 난이도(Lv1+Lv2) × 10 문제 = 400 문제
- 1 문제당 ~8 LLM 호출 (창작 4 + 검증 3 + 웹툰 1)
- 총 ~3,200 LLM 호출
- Sonnet 4.6 기준: **약 $20-40 (1회성 초기 투자)**

## 7. 학생(수지) UX

### 7.1 홈 화면
- 오늘의 미션 카드 (10문제 / 예상 시간 / 스트릭)
- 자유 연습 (단원 선택, 진도 %)
- 내 실력 (개념 마스터 수)

### 7.2 문제 풀이 화면 (B단계 = 개념 식별)
1. 문제 본문 표시
2. "이 문제는 어떤 개념?" 체크박스 리스트 (멀티셀렉트)
3. 제출 → 즉시 피드백

### 7.3 정답 화면
- ✅ 맞았어! + 정답 개념 표시
- 웹툰 대화체 해설 (왜 이 개념인지)
- [다음 문제]

### 7.4 오답 화면
- ❌ 아쉬워, 다시 볼까? (부드러운 톤)
- 선택 vs 정답 비교
- 웹툰 대화체 설명 ("~보다 크고 ~보다 작은"은 범위 시그널)
- 힌트 (선택적 노출)
- [비슷한 문제 1개 더] [다음]

### 7.5 세션 완료 화면
- 정답 수 / 시간 / 스트릭
- 개념별 정확도 (부드러운 표시, "약한 편이야" 정도)
- [홈으로] [한 세트 더]

### 7.6 UX 원칙
1. 모바일 퍼스트 (엄지 조작, 큰 버튼)
2. 5초 룰 (홈에서 5초 안에 시작)
3. 칭찬 우선 (오답도 "아쉬워" 톤)
4. 즉시 피드백 (매 문제 결과 바로)
5. 친구 같은 반말 톤 + 이모지
6. 힌트는 선택 (강요 X)

### 7.7 수지에게 절대 노출 안 할 데이터
- 구체적 정답률 숫자 (% 숫자)
- 오답 개념 리스트
- 학습 시간 통계
- → **부드러운 신호만** 노출 ("약한 편이야" 정도)

## 10. 보안

### 10.1 인증
- Supabase Auth (Email + Password)
- 세션: JWT, httpOnly 쿠키
- 아빠가 수지 계정 초기 생성, 초기 비번 전달 후 변경 유도

### 10.2 권한 (RLS)
- 수지: 자기 데이터만 읽기/쓰기
- 아빠(admin role): 모든 수지 데이터 읽기 + 문제/개념 쓰기
- 미인증: 모든 테이블 접근 금지

### 10.3 API 키 관리
- Vercel 환경변수 (웹앱)
- 아빠 로컬 `.env` gitignored (파이썬 워커)
- `.env.example` 공개

### 10.4 기타 방어
- HTTPS 강제
- CSP 헤더 (XSS 방어)
- Rate limiting (로그인 시도)
- Drizzle ORM 파라미터 바인딩 (SQL injection 방어)

### 10.5 저작권 안전장치
- PDF 원문 서버 업로드 금지 (아빠 로컬에만)
- `source_trace`: 참고 PDF 메타만 저장, 본문 X
- 생성 문제 = LLM 창작물 명시
- 워커 프롬프트에 "원문 인용 금지" 가드

### 10.6 감사 로그
- `problem_reviews` (검증 이력)
- `attempts` (수지 활동)
- 관리자 접속 로그 (선택)
